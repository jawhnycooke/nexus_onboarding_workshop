THE PAIN

Sync blocks. Async frees your process.

Call sync fit() and your Python process is tied up for minutes. If the network drops, you lose the job. You cannot run two experiments in parallel. You cannot kick off training before a meeting and score the model after.

Every production integration hits these walls eventually. Async is how you walk through them instead.

Async training.

submit returns in about a second. poll returns Self when done.
module_05_async_patterns.ipynb · cells 8 & 10 
nexus_async = NEXUSClassifier(mode="quality")

# submit — returns a task id immediately
fit_task_id = nexus_async.submit_fit_task(
    X_train_slim, y_train,
)
print(f"submitted: {fit_task_id}")  # save this

# poll — returns Self when done, None if still running
POLL_INTERVAL = 20
fitted = nexus_async.poll_fit_result(fit_task_id)
while fitted is None:
    time.sleep(POLL_INTERVAL)
    fitted = nexus_async.poll_fit_result(fit_task_id)

# nexus_async is now fitted — ready for predict, proba, feature importance
print(nexus_async.trained_model_id_)
WHY THIS MATTERS

Your process is free immediately.

The training job runs on Fundamental's infrastructure. submit returns in about a second. Everything between submit and the final poll is yours — other work, other jobs, or nothing at all.

# async probability prediction
proba_task = nexus_async.submit_predict_proba_task(X_holdout_slim)
proba = nexus_async.poll_predict_result(proba_task)
while proba is None:
    time.sleep(10)
    proba = nexus_async.poll_predict_result(proba_task)
# → np.ndarray shape (n_samples, n_classes)

# async class-label prediction — same poll method
labels_task = nexus_async.submit_predict_task(X_holdout_slim)
labels = nexus_async.poll_predict_result(labels_task)
while labels is None:
    time.sleep(10)
    labels = nexus_async.poll_predict_result(labels_task)
# → np.ndarray of int labels
WHY THIS MATTERS

One poll, two results.

Four submit verbs — fit, predict, predict_proba, feature importance — map to three poll methods. predict and predict_proba both retrieve through poll_predict_result: it returns class labels for a predict task and a probability matrix for a predict_proba task.

09 / 16

Submit both. Poll once.

N experiments. Wall-clock time of the longest.
module_05_async_patterns.ipynb · cells 17 & 18 
nexus_A = NEXUSClassifier(mode="quality")
nexus_B = NEXUSClassifier(mode="speed")

# submit both before polling either — neither blocks
task_A = nexus_A.submit_fit_task(X_train_slim, y_train)
task_B = nexus_B.submit_fit_task(X_train_full, y_train)

# one shared loop — checks each until done
done = {"A": False, "B": False}
while not all(done.values()):
    if not done["A"] and nexus_A.poll_fit_result(task_A) is not None:
        done["A"] = True
    if not done["B"] and nexus_B.poll_fit_result(task_B) is not None:
        done["B"] = True
    if not all(done.values()):
        time.sleep(20)
# both fitted once the loop exits — no sequencing required
WHY THIS MATTERS

The real multiplier.

Sequential, two training runs cost two wall-clocks. Parallel, they cost one. The only ceiling is the number of concurrent jobs your API tier supports — run as many experiments as you can submit.

THE MULTIPLIER

Two jobs. One wall-clock.

Sequential training runs cost you two wall-clocks. Parallel training — when you submit both jobs before polling either — costs you the longer of the two. N experiments for the price of the slowest.

Both jobs run on Fundamental's infra simultaneously; your shared poll loop just checks each. There is no orchestration to write on your side.

TASK PORTABILITY

Submit at night. Poll in the morning.

The submission and polling sides share nothing but the task id. A data pipeline at 11 PM submits training and saves the id. An evaluation script at 7 AM loads the id, creates a fresh NEXUSClassifier, and polls — on a different machine, in a different process, with no shared memory.

Once a job completes, its trained_model_id_ is the permanent handle. Pass it to load_model() on any estimator to reload the model — the same verb you learned in Module 02.

TWO IDS, TWO LIFETIMES

Save them both. Trust them differently.

task_id is the handle on an in-flight job. Use it to poll, to resume from another session, to pass between teams. After 48 hours it is evicted from the task registry — the job itself is long finished, but the handle is gone.

trained_model_id_ is the handle on the weights. It lives in the model registry permanently. Every future session that wants to predict starts by setting this id on a fresh estimator.

THE OPERATIONAL KEY

task_id:
48 hours
trained_model_id_:
forever

The moment a poll returns, you have both. Save the trained_model_id_ with care — that is what Modules 02 and 06 reload. The task id has done its job.

48h → ∞
one id fades,
the other stays.
