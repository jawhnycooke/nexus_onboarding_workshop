05 / 17

One function, every environment.

Per-env keys. Each environment maps to its own api_url. No code changes between dev and prod.
module_08_operating_at_scale.ipynb · cell 6 
ENV_API_URLS = {
    "dev":  "https://dev.api.fundamental.tech",
    "stg":  "https://stg.api.fundamental.tech",
    "prod": "https://api.fundamental.tech",
}

def build_client(env="dev"):
    suffix = env.upper()
    key = (os.getenv(f"FUNDAMENTAL_API_KEY_{suffix}")
           or os.getenv("FUNDAMENTAL_API_KEY"))
    if not key:
        raise ValueError(f"No API key for env: {env}")
    # each environment maps to its own endpoint
    api_url = ENV_API_URLS[env.lower()]
    client = Fundamental(api_key=key, api_url=api_url)
    fundamental.set_client(client)
    return client

# usage — same code, any environment
dev_client  = build_client(env="dev")
prod_client = build_client(env="prod")
WHY THIS MATTERS

Rotation is a variable update.

Key rotation, environment promotion, and staging deploys all become changes to environment variables — never to source. The estimator code from Modules 01 through 07 stays exactly the same.

model_id = nexus.trained_model_id_

client.models.set_attributes(model_id, {
    "version": "1.0",
    "owner":   "credit-risk-pod",
    "stage":   "production",
})

# read the metadata back from the model id
metadata = client.models.get(model_id)
print(metadata.attributes)
WHY THIS MATTERS

Metadata travels with the id.

set_attributes does not train anything — it tags the model the fit already produced. Version, owner, and stage stay attached to the model id, so future-you can identify it without reading commit history.

07 / 17
model_version EXPLAINED

Pin the model version.

By default a fit trains on the latest NEXUS model version, which can change over time. When a fit must be reproducible across a fleet — or you need to retrain a model exactly as it was — pin it with the model_version constructor parameter: NEXUSClassifier(mode="quality", model_version="0.1.7").

Leave it unset to track the latest. Call client.models.get_supported_versions() to list the available versions and the current default before you pin.

LOOK UP AND RETIRE

Inspect a model. Prune the stale ones.

client.models.get(model_id) returns a TrainedModelMetadata — the model id and the attributes dict you stamped on it. Read the stage attribute before you load a model into production; it catches the wrong-environment id at the boundary, not at first prediction.

client.models.delete(model_id) permanently removes a model. Gate it on the tags you set: only prune what is labelled an experiment, never a model marked production.

THE FOUR-METHOD SURFACE

get,
set_attributes,

Train it, tag it, list it, look it up, retire it. Every step in the model lifecycle is a single call on client.models — independent of the estimator that created the model.

Three habits. One lifecycle.

The client.models surface only pays off with discipline — tag on fit, look up on deploy, prune on retire.
01

Tag on fit

Every fit() is followed by set_attributes() — version, owner, stage. An untagged model is an orphan waiting to happen.

02

Look up on deploy

Before loading a model, call get(model_id) and verify stage is production. Catch the wrong-environment id at the boundary.

03

Prune on retire

list() the inventory, filter on the tags you set, delete() stale experiments. Production stays locked by its stage tag.

# reads FUNDAMENTAL_API_KEY / FUNDAMENTAL_API_URL from the env
client = Fundamental()
models = client.models.list()

# TrainedModelMetadata objects — not strings. Read id + attributes.
inventory = pd.DataFrame([{
    "model_id": m.trained_model_id,
    "version": m.attributes.get("version"),
    "owner":   m.attributes.get("owner"),
    "stage":   m.attributes.get("stage"),
} for m in models])

# audit is just pandas — sort, filter, groupby
prod = inventory[inventory["stage"] == "production"]
print(prod.to_string(index=False))
WHY THIS MATTERS

Model ids are production artifacts.

trained_model_id_ is the only handle you have on a trained model. models.list() is how you verify your registry is complete — compare periodically and find the orphans before they become incidents.

12 / 17
VALIDATED STARTUP

Fail at boot. Not at the first call.

build_validated_client wraps build_client with one extra step: call client.models.list() before returning. It is a fast, read-only API round-trip that proves the key is valid and the endpoint is reachable.

Pair with initialize_production(env, model_id): validate the client, load the trained_model_id_ from your registry, return a ready-to-predict estimator. Any auth or network failure raises before your service ever handles a real request.

THE LOAD-BEARING LINE

models.list() is cheap. Running it at startup turns silent rotation failures into loud startup errors — which is exactly what your on-call rotation wants.

THE DISCIPLINE

Write every model_id the moment fit returns.

A registry is a file — model_id, task, mode, created_at, dataset, notes — appended every time training completes. No exceptions. No "I'll write it after lunch." Every lost row is a future incident.

Reconcile against models.list() periodically. Anything in the inventory that is not in your registry is an orphan — a model someone forgot to record. Orphans do not cause incidents until someone tries to roll back to one.

THE CHECK

registry
+
models.list()
→ no orphans

Compare the two sets. Every model in production should be in both. Every experiment should be in the registry. Anything only in models.list() is a question you have not answered yet.

the ship-it invariant
